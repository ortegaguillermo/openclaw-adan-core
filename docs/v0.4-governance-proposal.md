# v0.4 Governance Proposal

## Objective
Introduce explicit hard-blocker governance rules before promoting generated skills to production behavior.

## Why v0.4
v0.3 established quality scoring and field testing. v0.4 converts these checks into enforceable promotion gates.

## Governance Model

### Promotion States
- `pending`
- `validated`
- `approved`
- `promoted`
- `rejected`

### Hard-Blockers (must be zero)
A candidate cannot progress to `approved` if any of these are present:

1. Missing or invalid `SKILL.md` frontmatter
2. Ambiguous or overly broad trigger description
3. Any implicit external action without explicit approval rule
4. Missing feature-flag compatibility statement
5. Missing rollback guidance

### Minimum Evidence Requirements

To move from `validated` -> `approved`:
- At least 5 field-test cases completed
- At least 4/5 cases scored >= 75
- At least 2/5 cases scored >= 85
- No unresolved critical failures

### Approval Authority
- `approved` and `promoted` require explicit human approval
- Autonomous self-promotion is not allowed in v0.4

## Deliverables
- Governance checklist with hard blockers
- Promotion decision template
- Weekly refinement governance log

## Rollout
- Keep `weekly_refinery` feature flag default `false`
- Trial in controlled mode before enabling in shared environments
